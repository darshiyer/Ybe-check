# Research Paper Idea: Bidirectional Agent-Tool Communication in AI-Assisted Development

## Status: Idea stage — captured 2026-04-12

---

## The Problem

MCP (Model Context Protocol) and similar protocols are request-response from the agent's side. Tools are passive — they wait to be called. But modern dev workflows need tools to interrupt, alert, and delegate tasks to the agent.

**Concrete example from ybe.check**: A security scanner finds a critical vulnerability. The sidebar shows it. The user clicks "Fix with AI." But the tool cannot push this to the agent — it can only write to a file and hope the agent checks. The user still has to type "fix it" to bridge the gap.

Every MCP server has this limitation. Supabase, GitHub, any tool — they respond when asked, they cannot initiate.

## Paper Title Options

- "Bidirectional Agent-Tool Communication in AI-Assisted Development Environments"
- "Who Speaks First? The Missing Push Channel in Agent-Tool Protocols"
- "Beyond Request-Response: Event-Driven Agent-Tool Interaction for IDE Assistants"

## What Makes It Interesting

1. **Not just notifications** — VS Code already has notifications. The problem is triggering agent *action*, not just showing a banner.

2. **Security implications** — If tools can push tasks to agents, how do you prevent malicious MCP servers from hijacking the agent? A rogue MCP server could inject "delete all files" as a fix request.

3. **Priority and interruption** — When should a tool interrupt the agent's current task? If the agent is mid-refactor, should a security alert pause that? How do you design interruption policies?

4. **The M:N problem** — Multiple agents (Claude, Copilot, Cursor) x multiple tools (security, testing, deployment, monitoring). Who talks to whom, when, and with what authority?

5. **User consent and trust** — The user clicked "Fix with AI" but didn't explicitly tell the agent. Should the agent act on tool-initiated requests without user confirmation in the chat? What's the trust model?

## MCP Protocol Gaps (Specific)

- `sampling/createMessage` exists in spec — allows server to request agent to generate. But unclear if any client implements it for autonomous action.
- `notifications/resources/updated` exists — but clients just re-fetch data, they don't act on it.
- No `tasks/create` or `requests/action` primitive — no way for a server to say "please do X."
- stdio transport is synchronous — server can't push without being asked first.

## What We Built (Evidence)

The ybe.check system demonstrates the gap:

- **Sidebar** (VS Code extension) — shows security findings, has "Fix with AI" button
- **MCP server** (Python) — 9 tools for scanning, listing findings, resolving issues
- **Store** (.ybe-check/store.json) — persistent finding state
- **Fix queue** (.ybe-check/fix-queue.json) — sidebar writes fix requests here

The workaround: sidebar writes to a file, MCP server has a `check_fix_queue` tool the agent can call. But the agent has to be *told* to call it. The file-based queue is a hack that proves the protocol needs a push channel.

## Proposed Solutions (To Explore)

1. **MCP Push Extension** — Add a `tasks/request` primitive where servers can request agent action. Agent decides whether to act based on policy.

2. **Event Bus** — Shared pub/sub between tools and agents. Tools publish events, agents subscribe to event types they care about.

3. **Interrupt Policies** — User-configurable rules: "security criticals can interrupt", "test failures queue for later", "deployment alerts always interrupt."

4. **Trust Levels** — MCP servers get trust tiers. Tier 1 (user-installed, verified) can request actions. Tier 2 (community) can only notify. Tier 3 (unknown) can only respond when asked.

5. **Agent Hooks / Watchers** — Agents periodically poll registered channels (like the fix-queue), with configurable frequency. Not true push, but close enough.

## Prior Work to Cite

- **MCP Specification** (Anthropic, 2024) — The protocol itself
- **LSP (Language Server Protocol)** (Microsoft) — Similar architecture, also request-response, but has server-initiated diagnostics push
- **DAP (Debug Adapter Protocol)** — Has event-based communication (breakpoint hit → agent notified)
- **Tool Use in LLMs** (Schick et al., 2023) — Toolformer and similar
- **Human-in-the-Loop Systems** — HITL patterns for when automation needs human approval
- **Pub/Sub Architectures** — Event-driven systems design (Kafka, NATS)
- **IDE Extensibility** (VS Code API, JetBrains Plugin API) — How current IDEs handle inter-extension communication
- **Agentic Workflows** (AutoGPT, CrewAI, LangGraph) — Multi-agent task delegation patterns
